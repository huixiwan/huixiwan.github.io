分布式训练

Megatron 与 3D 并行

之前讲述了ZeRO-123,和FSDP。但是前述的方法在处理特别大模型的时候还是会存在一些问题，前述方法本质还是数据并行，每张卡会计算完整的模型。当模型达到一定程度时，ZeRO-3的都要All-Gather参数值，通信量大，效率较低。因此可以换一个思路，直接切模型本身，也就是TP和PP。

1. TP (Tensor Parallelism) —— 张量并行

1.1 核心逻辑：把矩阵乘法拆开

TP 不是在层与层之间切分，而是在 层内部 把矩阵乘法（Matrix Multiplication）拆分到多张卡上并行计算。

1. MLP 层的切分（纵切 + 横切）

这是 Megatron 最精妙的地方，为了尽量减少通信，它采用了“两层一组”的策略：

  • 第一层（纵切/Column Parallel）：把权重矩阵 $W$ 按列切开。每张卡算出输出的一部分。
  • 第二层（横切/Row Parallel）：把权重矩阵 $W’$ 按行切开。每张卡直接拿上一层的局部输出进行计算。
  • 为什么这么做？：连续两个线性层如果用同一种切法，中间就必须通信。只有纵切+横切的组合能让第一层的输出直接作为第二层的输入，实现两层之间零通信。这是由矩阵运算的性质决定的，不是优化策略，而是唯一可行的方式

实际例子：
X = [x1 x2 …]
W1 = [A B]
W2 = [C]
[D]
X * W1 * W2 = [XA XB] * W2 = [XAC+XBD]
卡1: X * W1_Left * W2_top= [XAC]
卡2: X * W1_Right * W2_bottom = [XBD]
Reduce求和: [XAC+XBD]

因此（纵切 + 横切）需要进行一次通信

2. Attention 层的切分

  • 同样利用了多头注意力（Multi-Head Attention）的特性，把不同的 Head 分给不同的显卡。

1.2 物理直觉

1. 通信频率极高

  • 精确对比：
    • DP：一轮迭代（Step）仅在反向传播结束时通信 1 次（同步梯度）。
    • TP：在每个 Transformer Block 的前向和反向过程中都要通信。以 32 层模型为例，一轮迭代需要进行 128 次 All-Reduce 通信。
  • 结论：TP 的通信频率比 DP 高出两个数量级。这决定了 TP 必须部署在具有极高带宽的 机内 NVLink 环境下，跨机通信会成为严重的性能瓶颈。

2. 权重格式（Megatron 格式）

  • 现象：你会发现 Megatron 的模型文件（Checkpoint）和 HuggingFace 的不一样。
  • 原因：因为 TP 把权重矩阵物理上切碎了存到了不同的卡上。
  • 实战：在平台上部署时，经常需要一个 convert_checkpoint.py 脚本，把 HF 格式转换成 Megatron 格式（按 TP 维度切分），或者反过来。

3. 显存收益

  • TP 不仅切分了参数 $P$，还切分了 激活值 (A)。
  • 结论：如果你遇到了长文本导致的激活值 OOM，增加 TP 维度通常比增加 DP 维度更有效。
    这也是TP相较于ZeRO-3的优势，同时“切分了”激活值。

PP

PP 就是按层切模型，不同 GPU 负责不同的层，数据像流水线一样依次穿过各卡。
跟 TP 的区别：

  • TP：每张卡算同一层的不同部分，算完要通信合并
  • PP：每张卡算不同的层，算完把激活值传给下一张卡

简单示例，一个 4 层的模型切成 2 个 stage：
GPU0: Layer0 → Layer1 → 激活值传给GPU1
GPU1: Layer2 → Layer3 → 得到最终输出

通信特点：
1、计算之间有明显的先后顺序，因此PP会有气泡问题。
2、相邻卡之间为点对点通信。因此 PP可以跨机使用（InfiniBand 就行），这也是3D 并行中 PP 放在中间层、TP 必须在最内层的原因

气泡问题：
时间 →
GPU0: [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ ] [ ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ]
GPU1: [ ] [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ] [ ]
GPU2: [ ] [ ] [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ] [ ] [ ]
GPU3: [ ] [ ] [ ] [ F1 ] [ F2 ] [ F3 ] [ B1 ] [ B2 ] [ B3 ] [ ] [ ] [ ]

图解说明：
其中F1、F2、F3 就是第 1、2、3 个 micro-batch 的前向传播。
B1、B2、B3 就是micro-batch 的反向传播（注意：B1，B2，B3的顺序可以调整）
第一列是第一时刻，GPU0开始计算F1。
第二时刻，GPU0开始计算F2。GPU1收到GPU0的F1数据计算的激活值，开始计算。
当GPU0把F1，F2，F3都计算完毕后，便暂时无可计算数据。
当GPU3算完F3数据后，开始进行反向传播。逐个计算B1，B2，B3。
在反向传播过程中又会产生新的气泡。

解决方法：1F1B
核心思路：前向和反向交替进行，把气泡挤到首尾两端。
单卡示意图：
GPU0: [F1 F2 F3] [B1 F4] [B2 F5] [B3 F6] [B4 F7] [B5 F8] [B6 B7 B8]
^^^^^^^^ ^^^^^^^^ ^^^^^^^^ ^^^^^^^^ ^^^^^^^^
稳态阶段：1个反向 + 1个前向，交替进行，几乎无气泡
分三个阶段：

  1. 预热：先推几个前向把流水线填满（气泡在这） [F1 F2 F3]
  2. 稳态：每做 1 个前向紧接着做 1 个反向（几乎没有气泡） [B1 F4] [B2 F5] [B3 F6] [B4 F7] [B5 F8]
  3. 冷却：剩余的反向传播收尾（气泡在这） [B6 B7 B8]

两卡示例：
GPU0: [F1] [F2] [ ] [B1] [F3] [B2] [F4] [B3] [ ] [B4]
GPU1: [ ] [F1] [B1] [F2] [B2] [F3] [B3] [F4] [B4] [ ]

气泡只在首尾，中间的稳态阶段 GPU 基本没有空闲。micro-batch 越多，首尾气泡占比越小。

3D并行

3D 并行就是把 TP、PP、DP 三种并行嵌套套在一起用，同时训练一个超大模型。需要合理配置这三者的值。

嵌套顺序：
最外层：DP/ZeRO（跨机，All-Reduce，通信频率最低）
└─ 中间层：PP（跨机，相邻卡点对点，通信频率中等）
└─ 最内层：TP（单机内，NVLink，通信频率最高）

配置原则：
总 GPU 数 = TP × PP × DP，三个值相乘等于你拥有的卡数
TP 通信最密集，因此尽量在一台机子内，所以最大不超过8
TP 和 PP 能小就小，保证切分后的参数能存到最少的GPU内，剩下的全给 DP
TP 和 PP 两者的具体取值根据实际情况确定

举例，假设有 64 张卡（8 台机器，每台 8 卡）：
DP=4, PP=2, TP=8
总共需要：4 × 2 × 8 = 64 张卡
每份数据并行的副本 = PP=2 × TP=8 = 16 张卡

Megatron

就是使用3D并行训练的框架

1、准备数据集
Megatron 的数据格式跟 HuggingFace 不同。HuggingFace 通常读 JSONL 或 parquet，Megatron 要求转成二进制格式（numpy memory-mapped 文件）。

原因是：大模型训练时数据量很大（TB 级），JSONL 解析太慢，转成二进制后可以直接内存映射读取，IO 效率高很多。

Megatron 提供了转换脚本，类似：
python tools/preprocess_data.py –input my_data.jsonl –output_prefix my_data –tokenizer-type GPT2BPETokenizer
产出两个文件：.bin（tokenized 数据）和 .idx（索引文件）。

2、准备模型与模型转换
Megatron 的权重存储方式跟 HuggingFace 不一样。HF 是直接存完整的权重矩阵，Megatron 因为要做 TP切分，权重是按切分方式存储的。

所以如果要用 HF 上预训练好的模型（比如 LLaMA、Qwen），需要做一次格式转换。Megatron 提供了转换脚本，把 HF 的pytorch_model.bin 转成 Megatron 能识别的 checkpoint。

3、安装环境
最优：有一个已经集成的镜像。否则就只能自己做镜像了。
找一个合适的基础镜像，如NVIDIA 官方的 PyTorch 容器（包含 CUDA、NCCL 等底层库）
从 GitHub 拉取 Megatron-LM，pip 安装依赖（主要是版本兼容的问题——CUDA、PyTorch、NCCL、Megatron）
每个库安装完后，均需要做验证
过程中根据出现的bug进行修复。还有Megatron-LM能够找到一个稳定的commit记录，避免其更新产生依赖不兼容匹配。

4、编写训练程序
略。因为框架将大部分内容都已经封装了。而且往往会使用现成的基模型与数据集，只需在配置参数里配置好即可开始训练。

5、配置参数并运行启动脚本
Megatron 的参数主要通过命令行参数传入：

torchrun –nproc_per_node=8 pretrain_gpt.py
–tensor-model-parallel-size 8
–pipeline-model-parallel-size 2
–num-layers 32
–hidden-size 4096
–num-attention-heads 32
–seq-length 2048
–micro-batch-size 4
–global-batch-size 512
–train-iters 500000
–data-path my_data_text_document

其中 tensor-model-parallel-size 和 pipeline-model-parallel-size 就是控制 TP 和 PP
的，设好这两个值，框架自动处理切分和通信。多机多卡时用 torchrun 的 –nnodes 参数指定机器数。

实际可以将上述启动命令与配置参数写到启动脚本里，然后直接运行。
