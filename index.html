Ray

分布式计算框架，是目前RL和推理框架的主要依赖框架了

1.Ray核心架构

Driver:用户提交任务的入口
Head：运行了GCS（Global Control Service，全局控制服务），集群的大脑，用于监控全局的资源，元数据。
跑Raylet（本地调度器）和Plasma Store（共享内存对象存储）

  • Driver（用户代码）
    • 提交任务
      |
▼
  • Head 节点（纯 CPU 机器）
    • GCS（全局调度决策）
    • Raylet
    • Dashboard
      |
▼
  • Worker 节点 × N（一个服务器可运行多个 Worker）
    • Raylet（本地调度）
    • Plasma（共享内存对象存储）
    • Worker × N

架构说明：
head节点往往单独占用一台服务器，但是一个服务器可以同时运行多个worker
ray和k8s是独立的，所以上述均不必要是pod
head和worker实质都是一个python进程

核心组件说明：

  1. GCS只做调度决策和元数据管理，不执行计算
  2. Raylet是本地调度器，收到GCS的指令后在本地分配Worker执行任务
  3. Plasma Store是节点内的共享内存，同节点内的Worker之间传数据零拷贝。对比python原生多进程数据传输需要序列化，这是ray的一大优势

2.Ray的设计思想

ray是为了在python场景下方便地使用多进程能力设计的。
分布式计算采用多进程原因：多进程能跨机器，且GPU按进程分配，能实现资源隔离，方便Ray调度。
多进程通信难——》Plamsa Store。（同节点零拷贝共享内存）
多机器多进程管理困难——》GCS+Raylet的设计方式（全局调度 + 本地调度两层架构）
分布式场景容易故障——》故障恢复（Task重试、Actor Checkpoint、Lineage重建）

4.ray调度机制

4.1调度流程

1、ray会通过raylet将任务优先调度在自己节点
Driver 提交 Task → 本地 Raylet 检查资源足够 → 本地执行worker → Raylet 同步任务状态给GCS

2、如果当前节点的资源（CPU/GPU/内存）不满足任务需求，则交由GCS分析调度到其他节点
Driver 提交 Task → 本地 Raylet 检查资源不足够 → 请求 GCS 做全局调度决策

3、跨节点调度：优先调度到包含数据所在的节点（但是资源得足够）；如果所需数据分布在多个节点，则优先调度到数据量最大的节点，减少跨节点传输量。
GCS 考虑两个因素：资源需求（CPU/GPU）、数据本地性（输入数据在哪）→ GCS 通知目标节点的 Raylet → 分配 Worker 执行

4.2Placement Group（可以理解为批量预约资源，类上k8s的gang调度）

Ray 支持多种策略（类似 K8s 的亲和性）：

策略 行为 类比 K8s
PACK 尽量放同一节点 PodAffinity
SPREAD 尽量分散到不同节点 PodAntiAffinity
STRICT_PACK 必须放同一节点，放不下就失败 required PodAffinity
STRICT_SPREAD 必须分散，每个节点最多一个 required PodAntiAffinity

Placement Group 解决的是 Gang Scheduling 问题——保证一组 Actor 同时获得资源，不会出现”部分有资源、部分排队”的中间态。

RL 场景常用：多个 Rollout Actor 用 SPREAD（分散采样提高多样性），Rollout + Reward 用 PACK（放同节点减少跨节点传输）。

使用说明：
先定义Placement Group（bundles指定每组资源需求，strategy指定分布策略）
再通过.options(placement_group=pg)绑定到Actor/Task。

5.ray的故障恢复能力

故障类型 Ray 怎么处理
Task 挂了 自动重试（max_retries），在当前节点重新运行task
Actor 挂了 Checkpoint 恢复
GCS 挂了 Ray 2.x 支持 GCS HA（多副本 + Raft 选举）
Raylet 挂了 GCS 检测到节点失联，重新调度该节点上的 Task/Actor到其他节点。然后当前节点重新ray start连接上head节点
Plasma Store 挂了（OOM） 通过 Lineage（血统）重建丢失的对象
Head 挂了 Head 包含 GCS + Raylet，GCS 走 HA 恢复，Raylet 走节点故障逻辑
Worker 挂了 Ray 自动重新拉起 Worker 进程，通过ray start

下述仅需了解：

head节点重启

Head 节点上跑了三个东西：GCS、Raylet-Head、Dashboard。

  • Dashboard 挂了 → 无影响，只是看不到监控页面
  • Raylet-Head 挂了 → 和普通节点 Raylet 挂了一样，GCS 把上面的工作转移到其他节点
  • GCS 挂了 → 这是最严重的，因为 GCS 是集群大脑。Ray 2.x 之前直接集群不可用。Ray 2.x 引入 GCS HA：启多个 GCS 副本，用Raft 协议选 Leader，Leader 挂了自动切到备用副本（多个 GCS 副本都是 Head 节点上的进程，不是分散到 Worker 上的。本质上是 Head节点内多进程冗余，不是跨节点冗余。）

所以 Head 挂了本质上是拆开看它上面每个组件分别怎么处理。

Lineage（血统）重建：

Ray 会记录每个对象的”来路”：对象C是由 Task(对象A, 对象B) 产生的。这条依赖链就是 Lineage。

Plasma Store 里对象A、B、C都在内存里。如果 OOM 导致 C 被清掉了，Ray 发现 C 丢失了，就查 Lineage：C 依赖 A 和 B → A
和 B 还在 → 重新跑一遍产生 C 的那个 Task，C 就重建了。

类比：你知道一道菜的做法和原料都在，菜洒了，重新做一份就行。

GCS HA：

Ray 2.x 之前 GCS 是单点，挂了整个集群废掉。Ray 2.x 支持跑多个 GCS 副本，用 Raft
协议保证它们数据一致，一个挂了自动选新的 Leader 接管。

Actor的Checkpoint 恢复

Actor 在运行过程中，用户主动保存状态快照（Checkpoint），挂了之后 Ray 从最近的快照恢复。

流程：

  1. Actor 跑到第 300 步时挂了 → 自动从第 300 步的 checkpoint 恢复
  2. 丢失的是 300~挂掉时的进度，不会从头开始

关键点：checkpoint 是用户自己定义存什么和怎么恢复的，Ray
只负责调用。类似游戏的存档，你决定什么时候存、存什么，游戏挂了读档继续。

KubeRay

KubeRay 是 Ray 在 K8s 上的部署方案。
Ray 本身可以脱离 K8s 独立运行，但在生产环境中，需要 K8s 来管理 Pod的自动恢复、资源分配、故障转移（ray需要k8s的原因）。而k8s自身不具备对ray集群的管理能力，因此设计了kuberay插件，通过yaml实现ray集群的搭建销毁等管理（k8s需要KubeRay的原因）

1.整体架构

1.1 3种CRD

Ray Job
Ray Service
Ray Cluster

CRD 用途 生命周期
RayCluster 创建长期运行的 Ray 集群 手动删除才销毁
RayJob 提交训练 Job，跑完自动清理集群 Job 完成后自动销毁
RayService 部署 Ray Serve 在线推理服务 手动删除才销毁

1.2 整体架构

K8s 层：管理 Pod 的生命周期（创建、销毁、资源分配）
KubeRay：把 Ray 组件映射成 K8s 资源（GCS → Head Pod，Raylet → 每个 Pod）
Ray 层：管理 Task/Actor 调度、对象存储、故障恢复

│
├─→ 发现没有 Head Pod → 创建 Head Pod

实质就是先有k8s将用户创建资源信息存储到etcd。然后 K8s 自带的和第三方 Operator，他们的controller都会watch etcd，是kubeRay的controller能监听到kubeRay的资源，所以触发了KubeRay的调谐逻辑。而KubeRay的调谐逻辑，核心就是搭建ray集群，执行任务或是部署服务。

1.3 ray pod说明

Head Pod — 集群大脑，包含head相关组件+worker的组件：

  • GCS (Global Control Service)：全局元数据存储 + 全局调度（任务分发、Actor创建）
  • Dashboard：监控面板，暴露 8265 端口
  • Raylet：本地资源管理 + 任务执行（Head上的Raylet也跑任务，不只是管理） + 对象存储（Plasma）

Worker Pod — 计算节点，可多副本扩缩，包含：

  • Raylet：本地资源管理（CPU/GPU/内存注册）、任务调度、对象存储管理（Plasma）
  • Worker 进程：实际执行用户代码的进程，数量由 Pod 的 CPU 资源决定

注：Ray 2.x：Plasma 已经合并进 Raylet 进程内部，不再独立存在

Ray Node（逻辑概念） → Ray Pod（K8s物理载体），主要是生命周期交给了 K8s 管理

维度 Ray Node（裸机/Docker） Ray Pod（K8s）

扩缩容 手动 ray up 加机器 改 replicas 即可

3.KubeRay + Volcano 协作

3.1 为什么需要Volcano

K8s默认调度器逐个调度Pod，Ray集群的Head+Worker可能分散调度，部分Pod因资源不够Pending，已调度的Pod空跑等队友。

Volcano提供Gang Scheduling：要么所有Pod一起分配，要么全不分，避免资源碎片。

3.2 Volcano的组成

组件 类型 作用
volcano-scheduler 调度器（Pod） 替换default-scheduler，执行Gang/优先级/抢占调度
PodGroup CRD 声明哪些Pod是一组，minMember凑齐才一起调度
Queue CRD 队列管理，控制不同任务间的资源分配
VolcanoJob CRD 多角色任务模板（如MPIJob），自动创建Pod+PodGroup
  • volcano-scheduler + PodGroup：KubeRay场景只用这两层，调度器替换+调度约束
  • VolcanoJob：给没有专属Operator的框架用（如原生MPI），有KubeRay时不需要

3.3 KubeRay场景Volcano的使用方式

KubeRay仅使用Volcano的调度器。k8s支持指定调度器，因此只要在RayJob的yaml里指定k8s的调度器即可。即pod template里指定schedulerName: volcano，KubeRay自动创建PodGroup，不需要手动管理。

为什么要写这一组文章

大模型训练时，单卡显存往往装不下完整的参数、梯度和优化器状态；即便勉强装下，单卡算力也不够用。因此，分布式训练已经成为当前大模型训练的默认形态：多卡、多机协同，在通信、计算、存储三条线上同时做权衡。

训练基础

训练基础

本文只考虑神经网络的训练

1. 神经网络的物理本质（Infra 视角）

  • 本质：一系列张量（Tensor）的连续变换。数据以向量形式输入，经过层层矩阵运算，最终得到输出。
  • MLP 示例：最基础的结构 y = Wx + b。
    • 参数 (P)：即权重 $W$。Infra 侧关注其存储分布（单卡还是多卡）。
    • 激活值 (A)：计算过程中产生的中间变量。Infra 侧关注其显存占用，是前向传播的水位线。
    • 算子：如激活函数、Softmax。关注其计算开销与对显存碎片的贡献。

2. 训练的三大阶段与显存占用

2.1 显存账单核心成员

  • 参数 (P)：模型权重。如果以FP16格式，占用 $2N$ 字节。
  • 梯度 (G)：反向传播产生。如果以FP16格式，占用 $2N$ 字节。
  • 优化器状态 (O)：以 Adam 为例，包含一阶动量（历史梯度的滑动平均，加速收敛）、二阶动量（历史梯度平方的滑动平均，自适应调整学习率）、FP32 Master Weights（保存一份高精度参数，防止精度损失），共占用 $12N$ 字节（$4N$+$4N$+$4N$）。
  • 激活值 (A)：前向传播产生，随 Batch Size 和序列长度线性增长。

常见训练方式：

  • 纯 FP32 训练（老旧方法）：$P(4N) + G(4N) + O(8N) = 16N$。
  • 混合精度训练（主流）：$P(2N) + G(2N) + O(12N) = 16N$。

2.2 常见数据类型

数据类型 字节数 符号位 指数位 尾数位 备注
FP8 1 1 4 3 1 字节，极致压缩与推理场景，精度损失较大
FP16 2 1 5 10 2 字节，训练/推理常见，兼顾计算速度与精度
BF16 2 1 8 7 2 字节，动态范围与 FP32 相同（同 8 位指数），运算精度低于 FP16，但溢出风险更低，硬件适配好
FP32 4 1 8 23 4 字节，标准单精度浮点，与 BF16 取值范围一致，可无损转换为 BF16 参与快速计算
TF32 4 1 8 10 仅作为 TensorCore 内部计算格式，不用于显存存储。让 FP32 任务在不改代码的前提下获得近乎 FP16 的速度。

补充说明：

  • BF16 与 FP32 拥有完全一致的指数位（8 位），所以理论上 FP32 范围内的数都能“无溢出”直接转为 BF16 存储/计算，只损失尾数精度。PyTorch 等训练框架广泛支持 FP32↔BF16 自动转换，方便在硬件加速下运算。
  • TF32 (TensorFloat-32) 是 NVIDIA 新硬件上专为深度学习 TensorCore 设计的数据格式，精度介于 FP16 与 FP32 之间，加速大规模矩阵乘。
  • 选择数据类型时需权衡精度/动态范围/算力/内存，实际项目中常见混合精度与动态切换方案。

2.3 训练阶段显存占用情况

  1. 前向传播：水位 = $(P + O) + A$
    • 过程：随着层数加深，激活值 $A$ 像叠罗汉一样堆积。
    • 终态：到达前向传播结束，持有全量激活值。
  2. 反向传播：水位 = $(P + O) + A + G$
    • 过程：显存峰值在这个阶段。梯度 $G$ 在产生，激活值 $A$ 在逐层释放。
    • 峰值：出现在反向传播开始的瞬间，此时 $A$ 还没开始释放，而第一层梯度 $G$ 已经产生。
  3. 参数更新：水位 = $(P + O) + G$
    • 过程：此时 $A$ 已完全释放。优化器利用 $G$ 更新 $O$ 和 $P$。
    • 终态：更新完毕，清空 $G$。
  4. 休整期：水位 = $P + O$
    • 状态：显存回到最低水位，等待下一轮 Batch。

注：在第一轮迭代（Step 0）的起始点，显存仅占用 $P$；随着第一次 optimizer.step() 执行，优化器状态 $O$ 被初始化并从此常驻显存。

分布式训练优化策略与评估指标

分布式训练优化策略与评估指标

通过前文已经了解到由于单显卡无法承载大模型进行训练，所以引入了分布式训练。在分布式训练场景中，分布式通信是无法被忽视的一环，其对训练速度与稳定性会产生较大影响。
因此，在实现“能跑起来”之后，我们的核心目标就变成了：如何通过优化手段，尽可能掩盖或减少通信开销，从而压榨出硬件的极致算力。
本文将从评估指标、通信基础、优化策略三个维度，梳理分布式训练的性能调优思路。

1. 分布式训练核心指标

1.1 MFU (Model FLOPs Utilization) —— 最硬核指标

  • 定义：GPU 真正用于“模型数学运算”的时间比例，反映了对硬件算力的压榨程度。
  • 获取方法：无法直接读取，需根据公式 (6 * 参数量 * Tokens) / (StepTime * 显卡理论算力) 手动计算。
  • 实战意义：剔除了 IO 和通信的干扰，MFU 越高说明系统优化越好（标杆水平约为 50%）。

1.2 GPU 利用率 (GPU Utilization) —— 实时状态指标

  • 定义：GPU 在单位时间内处于活跃状态的比例。
  • 获取方法：nvidia-smi 或 watch -n 0.1 nvidia-smi。
  • 实战意义：理想状态是持续 90%+。若呈“锯齿状”波动，通常意味着存在数据加载（IO）或通信同步（Network）瓶颈。

1.3 GPU 显存利用率 (Memory Utilization) —— 显存水位指标

  • 定义：已分配显存占总显存的比例。
  • 获取方法：nvidia-smi。
  • 实战意义：
    • 显存占满不代表计算满：PyTorch 往往会预分配显存。
    • OOM 预警：若显存触顶而 GPU 利用率极低，说明静态显存占用过大，需考虑切分模型（ZeRO/FSDP）。

1.4 实际观测中的辅助指标

在生产环境中，除了上述核心指标，我们还会关注：

  • Step Time (步时)：衡量训练的稳定性，关注其波动情况。
  • Throughput (吞吐量)：衡量整体进度（Tokens/sec）。
  • 通信占比：通过 Profiler 观察 NCCL 通信耗时占总步时的比例，判断是否需要进一步做 Overlap 优化。
  • CPU/内存：判断是否存在数据预处理（DataLoader）瓶颈。

2. 分布式训练通信基础

2.1 为什么需要通信？

在分布式训练中，模型的状态（参数、梯度、优化器状态）或计算的中间结果被分布在不同的显卡上。为了保证训练的正确性和一致性，各卡之间必须进行数据交换：

  • 一致性要求：在数据并行中，各卡必须同步梯度，以保证更新后的模型参数完全一致。
  • 计算依赖：在模型并行（TP/PP）中，下一阶段的计算依赖于前一阶段在其他卡上的输出。

2.2 不同并行策略下的通信场景

  1. 数据并行 (DP/DDP)：

    • 内容：梯度 (Gradients)。
    • 时机：反向传播结束时，通过 All-Reduce 同步。

  2. 张量并行 (TP)：

    • 内容：激活值 (Activations) 的部分分量。
    • 时机：发生在每一层的 前向和反向计算过程中，通信频率极高。

  3. 流水并行 (PP)：

    • 内容：层与层之间的中间输出。
    • 时机：发生在 不同机器/阶段的交接处，数据像接力棒一样传递。

  4. ZeRO/FSDP：

    • 内容：模型参数 (P) 和 梯度 (G)。
    • 时机：计算前 All-Gather 收集参数，计算后 Reduce-Scatter 同步梯度。

2.3 通信原语简述

在分布式环境下，数据在不同节点间的流动主要依靠以下几种标准动作（原语）：

1. 基础动作（一对多 / 多对一）

  • Broadcast (广播)：一个节点将数据发给所有人。
    • [A] -> [A, A, A, A]
  • Scatter (散播)：一个节点将数据拆分，分别发给不同的人。
    • [ABCD] -> [A], [B], [C], [D]
  • Gather (聚合)：一个节点收集所有人的数据，拼成一份全量数据。
    • [A], [B], [C], [D] -> [ABCD]
  • Reduce (规约)：一个节点收集所有人的数据，并进行计算（如求和）。
    • [A], [B], [C], [D] -> [A+B+C+D]

2. 核心动作（多对多 - 分布式训练主力）

  • All-Gather：每个人都贡献一小块，最后每个人都拿到拼好的全量数据。
    • 场景：FSDP 前向计算前拉取全量参数。
    • [A], [B], [C], [D] -> [ABCD], [ABCD], [ABCD], [ABCD]
  • Reduce-Scatter：每个人都贡献一小块，先聚合计算，再把结果拆开，每人拿回一小块。
    • 场景：ZeRO-2/FSDP 反向传播后同步梯度并分摊存储。
    • [A], [B], [C], [D] -> [sum_A], [sum_B], [sum_C], [sum_D]
  • All-Reduce：每个人都贡献数据，最后每个人都拿到聚合（求和）后的全量结果。
    • 等价于：Reduce-Scatter + All-Gather。
    • 场景：DDP 模式下最通用的梯度同步手段。
    • [A], [B], [C], [D] -> [sum], [sum], [sum], [sum]

2.4 通信拓扑：从 Star 到 Ring

  1. Star 架构 (Parameter Server)：

    • 结构：所有计算节点（Worker）都找一个中心节点（Server）交换数据。
    • 瓶颈：当显卡变多时，中心节点的网络带宽会瞬间被挤爆，导致训练速度无法随显卡数量线性增加。

  2. Ring 架构 (Ring All-Reduce)：

    • 结构：显卡连成一个环，每张卡只和左右相邻的两张卡说话。
    • 优势：带宽利用率极高。无论你有 8 张卡还是 100 张卡，每张卡的通信压力是恒定的。
    • 结论：这是目前 PyTorch DDP 和主流大模型训练的标配，也是分布式系统能够大规模扩展的基础。

2.5 物理通信链路与分级同步

在真实集群中，通信效率受限于底层的物理连接。我们需要区分“节点内”和“节点间”两种链路：

1. 节点内通信 (Intra-node)

  • 物理链路：NVLink (NVIDIA) 或 HCCS (华为昇腾)。
  • 特点：带宽极高（如 H100 的 NVLink 可达 900 GB/s），延迟极低。
  • 策略：尽量在节点内完成高频的通信动作（如 TP 的同步）。

2. 跨节点通信 (Inter-node)

  • 物理链路：RDMA (RoCE / InfiniBand)。
  • 特点：带宽远低于节点内（如 400 Gbps，约 50 GB/s），但比普通以太网快得多。
  • 瓶颈：跨机通信是分布式训练的主要性能瓶颈。

3. 应对方案：分级同步 (Hierarchical Sync)

为了平衡带宽差异，现代通信库（如 NCCL）通常采用分级策略：

  • 第一步：在每台机器内部，利用 NVLink 快速聚合 8 张卡的梯度。
  • 第二步：每台机器只派出一个“代表”，通过 RDMA 网络与其它机器交换聚合后的结果。
  • 第三步：将最终结果在机器内部广播给所有卡。
  • 结论：这种方式极大地减少了跨机网络的数据包数量，保护了脆弱的跨机带宽。

3. 分布式训练优化手段

优化分布式训练的核心逻辑只有四个字：减量、藏时。可以从通信、计算等角度去优化。

3.1 减少通信数据量

  • 你的思路：尽可能每张卡存全量，避免同步。
  • 专业术语：混合精度训练 (Mixed Precision)。
  • 手段：使用 FP16 或 BF16 进行通信。
  • 效果：通信带宽直接减半。
  • 进阶：梯度压缩 (Gradient Compression)。通过量化或稀疏化减少传输的比特位。

3.2 降低通信频次

  • 你的思路：多跑几次迭代再汇聚。
  • 专业术语：梯度累积 (Gradient Accumulation)。
  • 手段：该部分手段较多：（1）设置 accumulation_steps=N，每跑 N 个 Batch 才进行一次 All-Reduce 同步。（2）通信合并，PyTorch DDP 默认会把多层梯度攒成一个 bucket一起传
  • 效果：将通信开销分摊到多个计算步中，显著提高计算/通信比。

3.3 通信掩盖 (Communication Overlap)

  • 你的思路：计算的同时提前传后续内容。
  • 专业术语：流水线并行 (Pipelining) 与 异步通信。
  • 手段：
    • 前向预取 (Prefetch)：在计算当前层时，后台异步拉取下一层的参数（如 FSDP 的核心优化）。
    • 反向掩盖：在计算当前层梯度时，后台异步同步已经算好的上一层梯度。
  • 本质：利用 GPU 的多流（Stream）特性，让计算和通信在时间轴上重叠。

3.4 算力压榨：算子融合 (Operator Fusion)

  • 背景：很多时候 GPU 忙着读写显存（Memory-bound），导致计算单元空转。
  • 手段：使用 FlashAttention 等融合算子。
  • 效果：减少了对显存的频繁访问，让 GPU 更多时间花在真正的数学运算上，从而间接提升了整体吞吐。

ZeRO-123和Deepspeed

前面的章节讲述了大模型训练的实现策略与分布式场景的优化。本章节开始讲述其中一种实现策略，ZeRO-123与Deepspeed

1. ZeRO原理

主要就是训练过程中的P+G+O无法被存储在一张显卡上，因此将P，G，O分配到其他卡上。

2. DDP 原始方案

ZeRO是在分布式训练基础上进行的，因此下述先简单讲讲分布式训练与单卡训练过程的不同点。
DDP就是最经典的分布式训练方案。

DDP 训练过程中，每张卡上都会加载同一份 P，各自使用不同的数据进行训练。每张卡算出的梯度 G 不同，因此需要通过All-Reduce 对 G 求平均，确保所有卡拿到相同的全局梯度后再更新 P，从而保证各卡上的 P 始终一致。

注意：这里是对G进行All-Reduce，而不是P。因为如果不对G求平均，使用不同的G不止更新出不同的P，还会更新出不同的O，这样的话，就要同步P和O了，而 O 的同步没有合理的公式。

DDP 的流程：

  1. 所有卡加载同一份 P，各自拿不同的数据
  2. 前向 + 反向 → 算出各自的局部梯度 G（不同）
  3. All-Reduce 同步 G → 每张卡拿到全局平均梯度 G（相同）
  4. 各卡用相同的 G 在本地更新 P → 更新后 P 自然相同

3. ZeRO-1

在 Adam 优化器中，O (优化器状态) 占比最大（12N，占静态显存的 75%）。ZeRO-1 发现 O 在前向和反向传播中完全不参与计算，仅在“参数更新”那一瞬间被访问。因此，将其均分到各卡上收益最高。

核心流程（以 8 卡为例）

  1. 前向传播：水位 = $(P + (1/8)O) + A$
    相比于单卡训练，前向传播过程中，O只占用了1/8，对应自己负责的那部分优化器状态。并且当前步骤并未使用到O，所以只存储(1/8)O不会对该过程造成影响
  2. 反向传播：水位 = $(P + (1/8)O) + A + G$
    同样当前步骤并未使用到O，所以只存储(1/8)O不会对该过程造成影响
    除此之外，A在反向传播过程中逐步释放（算完一层释放一层）。
  3. Reduce-Scatter：水位 = $(P + (1/8)O) + (1/8)G$
    这步是ZeRO-1增加的步骤。
    首先ZeRO-1是在DP的基础上进行训练的，所以需要进行Reduce操作，每部分的梯度都需要求平均。
    然后由于每张卡仅存储了(1/8)O，仅能更新对应的(1/8)P，因此每张卡上仅需要对应的(1/8)G就能完成对应的更新操作，所以仅需要Reduce-Scatter操作，让每张卡处理对应的(1/8)G即可，不需要All-Reduce。
  4. 参数更新：水位：更新前 $(P + (1/8)O) + (1/8)G$，更新后 $((1/8)P + (1/8)O)$
    由于每张卡只有(1/8)O，而切分策略保证了单卡上仅有的O能完成对应(1/8)P的更新，同时O也在同步更新。之后每张卡弃掉用不到的(7/8)P。
    同时在算完P后，G也会被释放。
  5. All-Gather：水位 = $P + (1/8)O$
    这步是ZeRO-1增加的步骤。
    为了后续快速的前向传播与后向传播计算，需要每张卡拥有完整的P，因此通过All-Gather将每张卡汇聚出完整的P

4. ZeRO-2

即使把优化器状态（O）均匀分布到所有显卡，每张卡依然无法容纳全部模型参数，因此还需要进一步优化显存：ZeRO-2 针对梯度（G）进行了切分。其核心思想是——反向传播计算过程中，每张卡只保留自身负责的 (1/8) G 分片，而不是完整保留所有梯度。

实现方式是：在反向传播进行到每一层时，都会同步执行一次 Reduce-Scatter 操作，把这一层产出的梯度立即切分、分发至各卡，确保每张卡始终只持有属于自己的那部分梯度分片 (1/8 G)。显著降低了单卡所需显存，从而支持更大规模的模型训练。

5. ZeRO-3

即使将O，G均分到其他卡上，仍放不下所有参数，甚至对于70B的模型，其模型参数量达到140GB，光模型参数一张卡就已经放不下了。因此需要将P也切分分配到其他卡上。

在ZeRO-3场景下，由于单卡缺少完整的参数，因此一张卡在正向传播过程中就需要进行通信了。
ZeRO-3中对每一层进行了切分，因此在每一层计算时均需要进行一次All-Gather完成该层的计算，然后在计算完后，丢弃同步过来的P。

核心流程：

  1. 前向传播：水位 = $((1/8)P + (1/8)O) + A$
    1.1 All-Gather当前层的参数
    1.2 计算当前层的激活值
    1.3 丢弃同步过来的层参数
    1.4 重复上述步骤直到全部层计算完毕

  2. 反向传播：水位 = $((1/8)P + (1/8)O) + A + (1/8)G$
    2.1 All-Gather当前层的参数
    2.2 计算当前层的梯度
    2.3 丢弃当前层的激活值与同步过来的层参数
    2.4 Reduce-Scatter当前层的梯度
    2.5 重复上述步骤直到全部层计算完毕

  3. 参数更新：水位：更新前 $((1/8)P + (1/8)O) + (1/8)G$，更新后 $((1/8)P + (1/8)O)$
    与ZeRO-1相同，各卡更新自己的P与O即可，更新完后，丢掉G

6. DeepSpeed

6.1 DeepSpeed 是什么

DeepSpeed 是微软开发的分布式训练框架，上面讲的 ZeRO-1/2/3 正是由 DeepSpeed 团队提出的。DeepSpeed 将 ZeRO 论文中的理论工程化，提供了一套易用的训练加速库。

简单理解：ZeRO 是论文/思想，DeepSpeed 是实现/框架。

6.2 核心能力

能力 说明
ZeRO-1/2/3 上面已详细讲述，核心显存优化手段
ZeRO-Infinity 将O/G甚至P卸载到CPU内存或NVMe磁盘，进一步突破显存瓶颈
3D并行 将数据并行（ZeRO）、张量并行、流水线并行组合使用
混合精度训练 FP16/BF16 + Loss Scaling，减少显存占用并加速计算
梯度累积 模拟大batch size，适合单卡显存不足以一次处理大批次数据的场景

6.3 ZeRO-Infinity

ZeRO-1/2/3 已经将显存分摊到多张GPU上，但当模型足够大（如 175B+），即使多卡分摊仍然放不下。ZeRO-Infinity 的思路是把不急需的数据暂时搬到更便宜但更慢的存储上：

  • 优化器状态 → 卸载到 CPU 内存（速度较快）
  • 参数/梯度 → 必要时卸载到 NVMe 磁盘（速度较慢，但容量大）

代价是训练速度会下降（CPU/NVMe 与 GPU 之间的数据搬运有延迟），但好处是用时间换空间，使得在有限GPU资源下训练超大模型成为可能。

FSDP

ZeRO-123能解决大模型的分布式训练问题，但该方法集成在DeepSpeed框架内。PyTorch为了解决大模型训练问题，在其内部集成了FSDP。因此使用FSDP的话，则不需要引入新的框架。

FSDP核心机制

FSDP = Fully Sharded Data Parallel，”Fully Sharded”的意思是 P、G、O 全部切分（PyTorch 原生版的ZeRO-3）。采用分片与重组机制，它将模型参数、梯度和优化器状态均匀分布在所有 GPU 上。本质是“用通讯换显存”。

FSDP的分片原理：
通过 auto_wrap_policy 标记需要分片的层（通常是 Transformer Block），将每层的参数展平后均匀切分到各卡上。未被标记的层同样展平切分。
Transformer模型的层结构规整，适合按层做统一的分片处理。

FSDP的重组原理：
先将未被标记的层 All-Gather 汇聚完整，然后逐层 All-Gather标记层的参数进行计算，算完立即丢弃，保证任意时刻只有一层是完整的

FSDP优化策略

CPU offload

问题：当模型较大时，在FSDP分片后，可能仍会出现(1/N)(P+G+O)在一张卡上放不下的情况。以 8B 模型 + 双卡 BF16 + AdamW 为例，单卡上 Adam 状态约 32GiB，加上参数和梯度，直接 OOM。

解决方法：CPU offload策略。（核心是让O一直存储在CPU侧，O的比重最大，能减轻GPU显存的压力）
P：前向阶段：CPU——》GPU（CPU侧复制一份到GPU侧，算完会丢弃），反向阶段：CPU——》GPU（CPU侧复制一份到GPU侧，算完会丢弃）
G：反向阶段：在GPU生成。Optimizer step：GPU——》CPU（直接移动）
O：一直在CPU侧

Activation Checkpointing

问题：当激活值占用较大的场合，即使能存储P+G+O，但仍可能由于A导致OOM。

解决方法：可以使用Activation Checkpointing方法。

正常情况:前向传播会保存每一层的激活值，供反向传播使用。
Activation Checkpointing：不保存中间层的激活值，只保存该层的输入张量。反向传播到该层时，从保存的输入重新执行前向计算，得到所需的激活值。

本质:以计算换显存。多花一遍前向计算的时间，省掉大量激活值的显存。

混合精度

为了加快计算速度，可以使用混合精度的方式来训练。参数在GPU以 BF16 存储（省显存），计算也用 BF16（GPU加速），优化器内部保留 FP32 主副本（用于更新P，保数值稳定）。计算用 BF16（快+省），更新用 FP32（稳），两者各取所长。
┌─────────────────┬──────────┬──────────────────────────────┐

分布式训练

Megatron 与 3D 并行

Megatron 与 3D 并行

之前讲述了ZeRO-123,和FSDP。但是前述的方法在处理特别大模型的时候还是会存在一些问题，前述方法本质还是数据并行，每张卡会计算完整的模型。当模型达到一定程度时，ZeRO-3的都要All-Gather参数值，通信量大，效率较低。因此可以换一个思路，直接切模型本身，也就是TP和PP。

1. TP (Tensor Parallelism) —— 张量并行

1.1 核心逻辑：把矩阵乘法拆开

TP 不是在层与层之间切分，而是在 层内部 把矩阵乘法（Matrix Multiplication）拆分到多张卡上并行计算。

1. MLP 层的切分（纵切 + 横切）

这是 Megatron 最精妙的地方，为了尽量减少通信，它采用了“两层一组”的策略：

  • 第一层（纵切/Column Parallel）：把权重矩阵 $W$ 按列切开。每张卡算出输出的一部分。
  • 第二层（横切/Row Parallel）：把权重矩阵 $W’$ 按行切开。每张卡直接拿上一层的局部输出进行计算。
  • 为什么这么做？：连续两个线性层如果用同一种切法，中间就必须通信。只有纵切+横切的组合能让第一层的输出直接作为第二层的输入，实现两层之间零通信。这是由矩阵运算的性质决定的，不是优化策略，而是唯一可行的方式

实际例子：
X = [x1 x2 …]
W1 = [A B]
W2 = [C]
[D]
X W1 W2 = [XA XB] W2 = [XAC+XBD]
卡1: X W1_Left W2_top= [XAC]
卡2: X W1_Right * W2_bottom = [XBD]
Reduce求和: [XAC+XBD]

因此（纵切 + 横切）需要进行一次通信

2. Attention 层的切分

  • 同样利用了多头注意力（Multi-Head Attention）的特性，把不同的 Head 分给不同的显卡。

1.2 物理直觉

1. 通信频率极高

  • 精确对比：
    • DP：一轮迭代（Step）仅在反向传播结束时通信 1 次（同步梯度）。
    • TP：在每个 Transformer Block 的前向和反向过程中都要通信。以 32 层模型为例，一轮迭代需要进行 128 次 All-Reduce 通信。
  • 结论：TP 的通信频率比 DP 高出两个数量级。这决定了 TP 必须部署在具有极高带宽的 机内 NVLink 环境下，跨机通信会成为严重的性能瓶颈。

2. 权重格式（Megatron 格式）

  • 现象：你会发现 Megatron 的模型文件（Checkpoint）和 HuggingFace 的不一样。
  • 原因：因为 TP 把权重矩阵物理上切碎了存到了不同的卡上。
  • 实战：在平台上部署时，经常需要一个 convert_checkpoint.py 脚本，把 HF 格式转换成 Megatron 格式（按 TP 维度切分），或者反过来。

3. 显存收益

  • TP 不仅切分了参数 $P$，还切分了 激活值 (A)。
  • 结论：如果你遇到了长文本导致的激活值 OOM，增加 TP 维度通常比增加 DP 维度更有效。
    这也是TP相较于ZeRO-3的优势，同时“切分了”激活值。

PP

PP 就是按层切模型，不同 GPU 负责不同的层，数据像流水线一样依次穿过各卡。
跟 TP 的区别：

  • TP：每张卡算同一层的不同部分，算完要通信合并
  • PP：每张卡算不同的层，算完把激活值传给下一张卡

简单示例，一个 4 层的模型切成 2 个 stage：
GPU0: Layer0 → Layer1 → 激活值传给GPU1
GPU1: Layer2 → Layer3 → 得到最终输出

通信特点：
1、计算之间有明显的先后顺序，因此PP会有气泡问题。
2、相邻卡之间为点对点通信。因此 PP可以跨机使用（InfiniBand 就行），这也是3D 并行中 PP 放在中间层、TP 必须在最内层的原因

气泡问题：
时间 →
GPU0: [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ ] [ ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ]
GPU1: [ ] [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ] [ ]
GPU2: [ ] [ ] [ F1 ] [ F2 ] [ F3 ] [ ] [ ] [ B1 ] [ B2 ] [ B3 ] [ ] [ ]
GPU3: [ ] [ ] [ ] [ F1 ] [ F2 ] [ F3 ] [ B1 ] [ B2 ] [ B3 ] [ ] [ ] [ ]

图解说明：
其中F1、F2、F3 就是第 1、2、3 个 micro-batch 的前向传播。
B1、B2、B3 就是micro-batch 的反向传播（注意：B1，B2，B3的顺序可以调整）
第一列是第一时刻，GPU0开始计算F1。
第二时刻，GPU0开始计算F2。GPU1收到GPU0的F1数据计算的激活值，开始计算。
当GPU0把F1，F2，F3都计算完毕后，便暂时无可计算数据。
当GPU3算完F3数据后，开始进行反向传播。逐个计算B1，B2，B3。
在反向传播过程中又会产生新的气泡。

解决方法：1F1B
核心思路：前向和反向交替进行，把气泡挤到首尾两端。
单卡示意图：
GPU0: [F1 F2 F3] [B1 F4] [B2 F5] [B3 F6] [B4 F7] [B5 F8] [B6 B7 B8]
^^^^^^^^ ^^^^^^^^ ^^^^^^^^ ^^^^^^^^ ^^^^^^^^
稳态阶段：1个反向 + 1个前向，交替进行，几乎无气泡
分三个阶段：

  1. 预热：先推几个前向把流水线填满（气泡在这） [F1 F2 F3]
  2. 稳态：每做 1 个前向紧接着做 1 个反向（几乎没有气泡） [B1 F4] [B2 F5] [B3 F6] [B4 F7] [B5 F8]
  3. 冷却：剩余的反向传播收尾（气泡在这） [B6 B7 B8]

两卡示例：
GPU0: [F1] [F2] [ ] [B1] [F3] [B2] [F4] [B3] [ ] [B4]
GPU1: [ ] [F1] [B1] [F2] [B2] [F3] [B3] [F4] [B4] [ ]

气泡只在首尾，中间的稳态阶段 GPU 基本没有空闲。micro-batch 越多，首尾气泡占比越小。

3D并行

3D 并行就是把 TP、PP、DP 三种并行嵌套套在一起用，同时训练一个超大模型。需要合理配置这三者的值。

嵌套顺序：
最外层：DP/ZeRO（跨机，All-Reduce，通信频率最低）
└─ 中间层：PP（跨机，相邻卡点对点，通信频率中等）
└─ 最内层：TP（单机内，NVLink，通信频率最高）

配置原则：
总 GPU 数 = TP × PP × DP，三个值相乘等于你拥有的卡数
TP 通信最密集，因此尽量在一台机子内，所以最大不超过8
TP 和 PP 能小就小，保证切分后的参数能存到最少的GPU内，剩下的全给 DP
TP 和 PP 两者的具体取值根据实际情况确定

举例，假设有 64 张卡（8 台机器，每台 8 卡）：
DP=4, PP=2, TP=8
总共需要：4 × 2 × 8 = 64 张卡
每份数据并行的副本 = PP=2 × TP=8 = 16 张卡

Megatron

就是使用3D并行训练的框架

1、准备数据集
Megatron 的数据格式跟 HuggingFace 不同。HuggingFace 通常读 JSONL 或 parquet，Megatron 要求转成二进制格式（numpy memory-mapped 文件）。

原因是：大模型训练时数据量很大（TB 级），JSONL 解析太慢，转成二进制后可以直接内存映射读取，IO 效率高很多。

Megatron 提供了转换脚本，类似：
python tools/preprocess_data.py —input my_data.jsonl —output_prefix my_data —tokenizer-type GPT2BPETokenizer
产出两个文件：.bin（tokenized 数据）和 .idx（索引文件）。

2、准备模型与模型转换
Megatron 的权重存储方式跟 HuggingFace 不一样。HF 是直接存完整的权重矩阵，Megatron 因为要做 TP切分，权重是按切分方式存储的。

所以如果要用 HF 上预训练好的模型（比如 LLaMA、Qwen），需要做一次格式转换。Megatron 提供了转换脚本，把 HF 的pytorch_model.bin 转成 Megatron 能识别的 checkpoint。

3、安装环境
最优：有一个已经集成的镜像。否则就只能自己做镜像了。
找一个合适的基础镜像，如NVIDIA 官方的 PyTorch 容器（包含 CUDA、NCCL 等底层库）
从 GitHub 拉取 Megatron-LM，pip 安装依赖（主要是版本兼容的问题——CUDA、PyTorch、NCCL、Megatron）
每个库安装完后，均需要做验证
过程中根据出现的bug进行修复。还有Megatron-LM能够找到一个稳定的commit记录，避免其更新产生依赖不兼容匹配。

4、编写训练程序
略。因为框架将大部分内容都已经封装了。而且往往会使用现成的基模型与数据集，只需在配置参数里配置好即可开始训练。

5、配置参数并运行启动脚本
Megatron 的参数主要通过命令行参数传入：

torchrun —nproc_per_node=8 pretrain_gpt.py \
—tensor-model-parallel-size 8 \
—pipeline-model-parallel-size 2 \
—num-layers 32 \
—hidden-size 4096 \
—num-attention-heads 32 \
—seq-length 2048 \
—micro-batch-size 4 \
—global-batch-size 512 \
—train-iters 500000 \
—data-path my_data_text_document

其中 tensor-model-parallel-size 和 pipeline-model-parallel-size 就是控制 TP 和 PP
的，设好这两个值，框架自动处理切分和通信。多机多卡时用 torchrun 的 —nnodes 参数指定机器数。

实际可以将上述启动命令与配置参数写到启动脚本里，然后直接运行。
