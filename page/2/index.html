分布式训练综合优化实战

分布式训练综合优化实战

本文将讲述几个具体模型的优化实战

环境说明

使用AutoDL平台
框架：FSDP
硬件：5090 x 2（32GB×2）

性能指标

tokens/s：每秒处理的 token 总数（双卡合计），直接反映训练吞吐。batch 越大、通信开销占比越小，tokens/s越高。
不选择 MFU 的原因：MFU 需要精确的理论 TFLOPS 和 FLOPs 乘数，这两者在实际环境中难以准确获取（sparse vs dense tensor core、模型架构差异等），算出来的值容易失真。tokens/s 是直接实测值，简单可靠。

Llama3-8B优化实战

模型分析

Llama3-8B，依据之前的显存账单计算公式：8*16 = 128 GiB
显存：32GB×2，显然存不下来。

第一阶段：能跑起来

初始状态：
BF16 + FSDP 全分片 + AdamW，FSDP 将 P+G+O 均分到 2 卡

核心瓶颈：
理论每卡约 64 GiB，但实际训练时 forward/backward 还需要临时显存（激活值、通信 buffer 等），因此在 optimizer.step() 之前就已经 OOM。

解决方法：
CPU offload

实测数据：
开启 CPU offload 后 P 和 O 搬到 CPU，GPU主要承担计算，峰值降到 15.5GiB。

系统资源利用率（mb=1, offload=true）：

┌──────────────┬──────────┐
│ 指标 │ 数值 │
├──────────────┼──────────┤
│ GPU 利用率 │ ~54% │
│ CPU 利用率 │ ~20% │
│ 内存占用 │ ~17% │
└──────────────┴──────────┘

GPU 利用率仅 54%，说明大量时间花在 CPU↔GPU 数据搬运上，GPU 处于等待状态。这也是后续优化的空间所在。

第二阶段：性能优化调试

方法：抬 throughput（固定 seq=512, offload=true）

在 offload 约束下扫 micro-batch：

┌─────────────┬──────────┬──────────┬──────┐
│ micro_batch │ tokens/s │ 峰值显存 │ 结果 │
├─────────────┼──────────┼──────────┼──────┤
│ 1 │ ~165 │ 15.5 GiB │ 基线 │
├─────────────┼──────────┼──────────┼──────┤
│ 2 │ ~307 │ 15.7 GiB │ │
├─────────────┼──────────┼──────────┼──────┤
│ 4 │ ~584 │ 19.3 GiB │ │
├─────────────┼──────────┼──────────┼──────┤
│ 6 │ ~875 │ 27.2 GiB │ │
├─────────────┼──────────┼──────────┼──────┤
│ 7 │ — │ — │ OOM │
└─────────────┴──────────┴──────────┴──────┘

从 mb=1 到 mb=6，throughput 提升约 5.3 倍。但 mb=7 时显存触顶（27.2 GiB 已接近 32GB 上限，加上通信 buffer 就 OOM了）。

图解（tokens/s 与显存随 mb 变化）：

随着 mb 增大，tokens/s 近线性增长，显存也同步上升。mb 越大，每次 GPU 计算量越充分，通信开销占比越小，吞吐效率越高。

第三阶段：用 AC 性能优化

mb 上不去，尝试 activation checkpointing + 降低 seq_len 换显存，再抬 mb：

┌────────────────┬─────────────┬──────────┬──────────┐
│ 配置 │ micro_batch │ tokens/s │ 峰值显存 │
├────────────────┼─────────────┼──────────┼──────────┤
│ seq=512, 无 AC │ mb=6 │ ~875 │ 27.2 GiB │
├────────────────┼─────────────┼──────────┼──────────┤
│ seq=512, 无 AC │ mb=7 │ — │ OOM │
├────────────────┼─────────────┼──────────┼──────────┤
│ seq=448, 有 AC │ mb=6 │ ~801 │ 16.0 GiB │
├────────────────┼─────────────┼──────────┼──────────┤
│ seq=448, 有 AC │ mb=7 │ ~936 │ 16.1 GiB │
├────────────────┼─────────────┼──────────┼──────────┤
│ seq=448, 有 AC │ mb=8 │ ~1068 │ 16.2 GiB │
└────────────────┴─────────────┴──────────┴──────────┘

AC 把激活值从 776MB 降到 264MB，省出来的显存刚好够抬一步 mb。最终 seq=448 + AC + mb=8 达到 >1000 tok/s。

AC 牺牲了少量 throughput（以计算换显存），但换来了 mb 的提升空间，最终综合吞吐反而更高。

总结

8B 在双卡 32GB 上的优化路径：

OOM → offload（能跑）→ 扫 mb（5.3x）→ 加 AC（突破 mb 瓶颈，>1000 tok/s）

瓶颈变化：显存不够 → offload 后通信/CPU 成为瓶颈 → AC 省显存后继续抬 batch

Qwen-3B优化实战

模型分析

Qwen2.5-3B，BF16 + AdamW。

与 8B 的核心差异：3B 的 P+G+O 在双卡 no_shard（DDP）模式下可以直接放下（基线峰值 ~27 GiB），不需要 CPU offload。8B是”在显存夹缝里求生”，3B 是”显存充裕，优化重点在算子层面”。

第一阶段：基线跑通

配置：no_shard + sdpa + fused_adam + tf32，offload=false

直接跑通，无需 offload。基线：mb=1, seq=512，~1734 tok/s，峰值 26.75 GiB。

第二阶段：扫 micro-batch 提吞吐

┌─────────────┬──────────┬──────────┬──────┐
│ micro_batch │ tokens/s │ 峰值显存 │ 结果 │
├─────────────┼──────────┼──────────┼──────┤
│ 1 │ ~1734 │ 26.75 GiB│ 基线 │
├─────────────┼──────────┼──────────┼──────┤
│ 2 │ ~3172 │ 29.85 GiB│ │
├─────────────┼──────────┼──────────┼──────┤
│ ≥4 │ — │ — │ OOM │
└─────────────┴──────────┴──────────┴──────┘

mb=1→2：throughput 提升 ~1.8x，显存增加 ~3 GiB。mb=4 即 OOM，3B 虽然不需要 offload，但单卡 32GB 放下完整模型后剩余空间有限，mb 上限很低。

图解（tokens/s 随 mb 变化）：

与 8B 优化思路对比

┌────────────────┬───────────────────┬───────────────────┐
│ 维度 │ 8B（双卡32GB） │ 3B（双卡32GB） │
├────────────────┼───────────────────┼───────────────────┤
│ 初始状态 │ OOM │ 直接能跑 │
├────────────────┼───────────────────┼───────────────────┤
│ 是否需要 offload│ 必须 │ 不需要 │
├────────────────┼───────────────────┼───────────────────┤
│ 优化路径 │ offload→扫mb→AC │ 直接扫mb │
├────────────────┼───────────────────┼───────────────────┤
│ 最高 tokens/s │ ~1068 │ ~3172 │
├────────────────┼───────────────────┼───────────────────┤
│ 瓶颈变化 │ 显存→CPU通信→显存 │ 显存（mb上限） │
└────────────────┴───────────────────┴───────────────────┘

总结

3B 优化路径：直接跑 → 扫 mb（~1.8x）。

全流程 tokens/s 演进：

相比 8B 需要多步（offload → 扫mb → AC → 调seq），3B 只需一步就触到天花板。原因很简单：显存够用，不需要 offload 和 AC 这类”以XX换显存”的手段，但 mb 上限仍然受限于激活值显存（no_shard 下每卡放完整模型，剩余空间有限）。要继续提吞吐，需要从算子层面入手（torch.compile、FlashAttention），但这些在当前环境下存在兼容性问题。
